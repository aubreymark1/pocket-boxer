# DEV_ROADMAP.md

《Pocket Boxer》项目开发推进流程

## 1. 总原则

本项目是黑客松项目，开发推进必须遵守一个核心原则：

先做出能玩的最小闭环，再扩展对战、动画和视觉效果。

不要一开始就追求复杂动作识别、在线 PVP、华丽 UI、完整音效系统。

项目最低成功标准是：

打开作品  
进入首页  
点击出拳测试  
完成倒计时  
用户挥动手机或使用备用蓄力模式  
系统计算出拳指数  
显示结果页  
进入人机对战  
完成三回合  
显示 Win / Lose / Draw

只要这条链路稳定，项目核心创意就成立。

## 5. 阶段2：验证手机能力

### 目标

先确认手机能力能不能用，不要先写复杂游戏。

### 负责人

传感器 / 游戏逻辑负责人

### 需要验证的能力

| 能力 | 优先级 | 是否必须 |
|---|---|---|
| 动作传感器权限请求 | P0 | 必须 |
| 加速度读取 | P0 | 必须 |
| 挥拳峰值识别 | P0 | 必须 |
| 震动反馈 | P1 | 建议 |
| Web Audio 基础音效 | P1 | 建议 |
| localStorage 存储 | P1 | 建议 |
| 备用蓄力模式 | P0 | 必须 |

### 开发任务

#### 任务1：传感器权限测试

实现一个临时测试页：

点击按钮  
请求 `DeviceMotionEvent` 权限  
显示授权成功或失败状态

验收标准：

手机上可以触发权限请求  
授权成功后能继续采样  
权限被拒绝时能进入备用模式

#### 任务2：加速度数据测试

实现 debug 面板，显示当前加速度数据：

`x`  
`y`  
`z`  
`magnitude`

验收标准：

手机移动时数据明显变化  
短幅度挥拳时峰值明显上升  
数据刷新稳定

#### 任务3：挥拳识别

实现：

静止校准  
倒计时后开启采样窗口  
记录峰值  
根据峰值计算分数

验收标准：

明显挥拳可以触发分数  
轻微晃动不会误触发高分  
一次挥拳不会被连续识别多次

#### 任务4：备用蓄力模式

实现：

长按屏幕蓄力  
松手出拳  
计算 0 到 100 分

验收标准：

桌面浏览器可演示  
传感器不可用时仍可继续流程  
结果页与战斗系统可复用同一套输出结构

### 产物

`src/motion.js`  
`src/feedback.js`  
`src/constants.js` 或阈值配置文件  
Debug 面板或临时测试页

### 阶段验收标准

至少完成传感器读取 + 挥拳分数 + 备用模式  
有 debug 输出  
真机可测  
阈值可以快速调整

## 7. 阶段4：扩展人机对战 MVP

### 目标

在出拳测试稳定后，扩展到人机三回合对战。

注意：

不要在出拳测试不稳定时做复杂战斗表现。

### 模式结构

| 模式 | 内容 | 优先级 |
|---|---|---|
| 模式一 | 出拳测试 | P0 |
| 模式二 | 人机对战 | P0 |
| 模式三 | PVP 概念页 | P1 |

### 人机对战规则

对战双方：

玩家  
拳击机器人

初始血量：

玩家 HP：100  
机器人 HP：100

每轮流程：

玩家出拳  
系统读取出拳指数  
玩家对机器人造成伤害  
机器人自动反击  
进入下一轮

玩家伤害：

`玩家伤害 = 出拳指数 * 0.5`

机器人伤害：

`机器人伤害 = 玩家伤害 * 0.8 ~ 1.2 随机系数`

胜负规则：

最多 3 轮  
任一方 HP 小于等于 0 则提前结束  
三轮后剩余 HP 更高者获胜  
相同则平局

### 开发任务

#### 任务1：战斗状态结构

实现：

回合数  
玩家 HP  
机器人 HP  
每回合伤害  
战斗结束状态

#### 任务2：战斗流程联通

实现：

从首页进入战斗准备页  
开始战斗  
每回合调用出拳流程  
出拳结果转换为伤害  
进入下一回合  
最终进入结算页

#### 任务3：结果展示

实现：

Win / Lose / Draw  
玩家剩余 HP  
机器人剩余 HP  
总伤害  
最高出拳指数

### 产物

`src/battle.js`  
战斗准备页  
战斗页  
战斗结果页

### 验收标准

能完整打完三回合  
玩家伤害计算正确  
机器人反击正常  
胜负判断正确  
战斗结果可视化清晰

## 10. 阶段7：演示版本冻结

### 目标

避免演示前最后时刻把项目改崩。

建议在正式演示前至少 2 小时冻结。

只允许修：

出拳测试无法运行

### 负责人

9. 展示战斗结算  
10. 补充说明后续 PVP 扩展空间

### 30 秒介绍稿

我们做的是一款手机体感拳击游戏，叫《Pocket Boxer》。玩家不是点击按钮出拳，而是握住手机，真实挥出一拳，系统会根据手机传感器数据计算出拳指数，再把它转化为战斗伤害。这个项目的重点不是做复杂格斗，而是把手机本身变成拳击手套，让玩家的真实动作直接影响游戏结果。

## 14. Bug 优先级规则

### P0：必须立刻修

包括：

页面无法打开  
首页无法进入测试  
传感器和备用模式都不可用  
结果页不显示  
战斗流程断掉  
真机无法演示

### P1：尽量修

包括：

传感器阈值不稳定  
机器人伤害偶尔异常  
音效偶尔不播放  
UI 有轻微错位  
震动偶尔不触发  
localStorage 不稳定

### P2：可以不修

包括：

动画不够精致  
PVP 页面没做  
称号不够丰富  
视觉细节不统一  
不是所有手机都适配得很好

## 15. 项目决策规则

当团队出现分歧时，按以下顺序决策：

1. 是否影响出拳测试核心闭环？  
2. 是否影响现场演示稳定性？  
3. 是否能在当前时间内完成？  
4. 是否需要多人同时改同一文件？  
5. 是否比现有方案更简单？

如果答案不明确，默认不做。
